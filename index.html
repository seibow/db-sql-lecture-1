データベースとは？

💬 「データベース」と聞いて、どんなものを思い浮かべますか？
データベースとは

広い意味では、データを整理してまとめたものはすべて「データベース」と言えます。

📚

図書館の蔵書

書名・著者・分類で整理された本の集合

📒

アドレス帳

名前・住所・電話番号が整理されたリスト

🛒

ECサイトの商品情報

商品名・価格・在庫数を管理するシステム

データベースにはいくつかの種類がありますが、現在もっとも広く使われているのがリレーショナルデータベース（RDB）です。データをテーブル（表）形式で管理し、表と表を「関係」で結ぶ仕組みです。

本講義では以降、「データベース（DB）」と言ったらRDBのことを指します。

📌 本講義では「データベース（DB）」＝ リレーショナルデータベース（RDB） を指します。

📋 テーブルの見方

RDBはデータをテーブル（表）として管理します。テーブルの構造を押さえましょう。

テーブル
顧客ID 氏名 都道府県 登録日
1001 山田 太郎 東京都 2024-01-15
1002 鈴木 花子 大阪府 2024-02-03
1003 田中 一郎 北海道 2024-03-22
カラム（Column） ＝ データの種類を表す縦の項目 レコード / 行（Row） ＝ 1件分のデータ フィールド（Field） ＝ カラムとレコードの交点にある1つの値

ファイル・Excelによる管理との違い

💬 データを管理するだけなら、Excelやファイルで十分ではないでしょうか？
ファイル・Excelとデータベースの違い
観点Excelやファイル管理データベース（RDB）
一元管理 ファイルが分散しやすく、どれが最新か分からなくなります データを1か所に集約して管理でき、常に最新の状態を保てます
大量データへの対応 数十万行を超えると動作が重くなり、クラッシュのリスクもあります 数百万・数億件のデータも安定して高速に処理できます
データの独立性 保存形式が変わると、利用するプログラムもすべて修正が必要になります データの保存方法とアプリケーションを切り離して管理できます
データ整合性 入力ミスや重複、矛盾したデータが入っても検知できません 制約（ルール）を設定することで、不正なデータの登録をDBが防いでくれます

主キーと外部キー

🔑 主キー（Primary Key / PK）

テーブル内の 各レコードを一意に識別する 列のことです。

  • 同じ値が2件以上存在してはいけません（一意性）
  • 空（NULL）にしてはいけません（必須性）
💬 「氏名や都道府県を主キーにしてはいけない理由は何でしょうか？」
主キーにできない理由

氏名は同名の人が複数いる可能性があります。都道府県は47種類しかないため、明らかに重複します。主キーには絶対に重複しない値を割り当てる必要があります。そのため、システムが自動採番する「ID」を使うのが一般的です。

🔑 顧客ID（PK） 氏名 都道府県 登録日
1001 山田 太郎東京都2024-01-15
1002 鈴木 花子大阪府2024-02-03
1003 田中 一郎北海道2024-03-22

「顧客ID = 1001」と指定すれば、山田太郎さんのレコードを必ず一意に特定できます。

🔗 外部キー（Foreign Key / FK）

別のテーブルの主キーを参照する列のことです。これによってテーブル同士の「関係（リレーション）」を表現します。

正規化とは？

正規化とは、データの重複や矛盾を排除するためにテーブルを整理・分割するプロセスです。「1つの事実は1か所に」が基本原則です。

前
正規化前（問題のあるテーブル）
注文ID顧客名注文商品（複数）金額（複数）
5001山田 太郎ノートPC, マウス¥80,000, ¥3,500
5002鈴木 花子キーボード¥12,000
⚠️ 1つのセルに「ノートPC, マウス」と複数の値が入っています。「マウスだけ注文した人を検索する」といった操作が非常に困難になります。
1
第1正規化：1つのカラムには1つの値だけを入れる

1セルに複数の値が入っている場合は、行を分割して1件ずつにします。

注文ID顧客名注文商品金額
5001山田 太郎ノートPC¥80,000
5001山田 太郎マウス¥3,500
5002鈴木 花子キーボード¥12,000
⚠️ 行を分けたことで「山田太郎」という値が2行に重複しました。山田さんの名前を変更する場合、全行を探して修正しなければならず、修正漏れが起きやすくなります。
2
第2正規化：主キーだけで決まる情報は別テーブルに分ける

「顧客名」は注文商品に関係なく顧客IDだけで決まる情報です。注文テーブルから切り出して顧客テーブルに移します。

顧客テーブル

顧客ID（PK）顧客名
C01山田 太郎
C02鈴木 花子

注文テーブル

注文ID（PK）顧客ID（FK）商品金額
5001-AC01ノートPC¥80,000
5001-BC01マウス¥3,500
5002-AC02キーボード¥12,000
⚠️ 注文テーブルに「商品」と「金額」が並んでいますが、金額は商品だけで決まる情報です（どの注文でもノートPCは¥80,000）。これも分離する必要があります。
3
第3正規化：主キー以外の項目に依存する情報を別テーブルに分ける

「金額」は主キー（注文ID）ではなく「商品」に依存しています。商品テーブルとして切り出します。

顧客テーブル

顧客ID顧客名
C01山田 太郎
C02鈴木 花子

注文テーブル

注文ID顧客ID商品ID
5001-AC01P01
5001-BC01P02
5002-AC02P03

商品テーブル

商品ID商品名金額
P01ノートPC¥80,000
P02マウス¥3,500
P03キーボード¥12,000
✅ 各テーブルの情報がきれいに分離されました。商品の価格が変わっても商品テーブルの1行を直すだけでOKです。「1つの事実を1か所に」が実現されています。
📌 正規化は「第3正規形まで対応できていれば実務上は十分」と言われることが多いです。設計の際に意識する基本的な考え方として覚えておきましょう。

SQLとは？

💬 DBMSに対してデータベースを操作するには、どうやって「命令」を伝えるのでしょうか？
SQLとクエリについて

人間がDBMSに命令を伝えるための専用の言語が必要です。それがSQLです。英語の命令文に近い構造を持っており、「どのデータを」「どうしたいか」を記述します。

そのSQLで書いた命令文のことをクエリ（Query）と呼びます。「問い合わせ」という意味の英単語です。現場では「このクエリ重いね」「クエリ投げてみて」のように日常的に使われます。

SQL とは

SQL（Structured Query Language / 構造化問い合わせ言語） は、DBMSを通じてデータベースを操作するための命令言語です。

🖥️ アプリケーション
（Webサービス・業務システムなど）
⬇️
SQL（クエリ）で命令を書く
⚙️ DBMS
（MySQL・PostgreSQL など）
⬇️
命令を解釈してデータを操作する
🗄️ データベース
（実際のデータが格納されている場所）

SQLでできること（4つの基本操作）

操作SQL文意味
取得（読み取り）SELECTデータを検索・取り出す
登録（追加）INSERT新しいデータを追加する
更新UPDATE既存のデータを変更する
削除DELETEデータを削除する
✅ 具体的なSQLの書き方は次回の講義で学びます。今日はまず「SQLとはDBMSへの命令言語である」というイメージをしっかり持っておきましょう。
